O que é o PyTest?

O PyTest é um framework de testes para Python — descobre testes automaticamente, organiza fixtures reutilizáveis e produz relatórios claros de falha.

  • Convenção: arquivos test_*.py, classes Test*, funções test_*
  • Fixtures — setup/teardown com injeção de dependência por nome do parâmetro
  • Markers — @pytest.mark.smoke, regression, api, visual…
  • Ecossistema de plugins: pytest-playwright, pytest-xdist, axe-playwright-python
  • Neste projeto: PyTest orquestra · Playwright executa browser e HTTP
Diferente de unittest, o PyTest não exige herança de classes base — basta funções test_* e assert puro com introspection rica nas falhas.

Por que PyTest + Playwright?

Python nativo

Ideal para times QA/dev que preferem Python a TypeScript.

Fixtures declarativas

autouse, scope="session" — auth cache uma vez por run.

E2E + API unificados

page para UI e api_request para contratos REST.

PyTest — assert introspection

Diferencial #1 para QAs: falhas legíveis sem boilerplate de matchers customizados.

def test_kpi_runs_is_positive(page):
    text = DashboardPage(page).kpi_value("runs").inner_text()
    assert int(text) > 0          # pytest mostra valores reais
    assert "Demo User" in greeting  # diff de strings no terminal

users = [{"role": "admin"}, {"role": "viewer"}]
assert users[0]["role"] == "viewer" # AssertionError + diff side-by-side
  • assert puro — sem API de assertion do framework
  • Relatório mostra left/right em dicts, listas e strings
  • Combine com expect() do Playwright para UI com auto-wait

@pytest.fixture(scope="session", autouse=True)
def cache_auth_token(playwright) -> str:
    api_ctx = playwright.request.new_context(base_url=BASE_URL)
    session = fetch_auth_token(api_ctx)
    write_token_cache(session["token"], session["email"])
    return session["token"]

@pytest.fixture
def api_request(playwright) -> APIRequestContext:
    ctx = playwright.request.new_context(base_url=BASE_URL)
    yield ctx
    ctx.dispose()

Page Object Model (POM)

Specs finos · Page Objects encapsulam o DOM · retorno Self para method chaining.

  • Test spec — cenário de negócio, Given/When/Then, fixtures
  • Page Object — locators, ações, assertions

Anatomia — LoginPage

class LoginPage:
    def email_input(self) -> Locator:
        return self.page.get_by_test_id("login-email")

    def login_with(self, email: str, password: str) -> Self:
        self.fill_email(email)
        self.fill_password(password)
        self.submit()
        return self

    def should_redirect_to_dashboard(self) -> Self:
        expect(self.page).to_have_url(re.compile(r"/web/dashboard\.html"))
        expect(self.page.get_by_test_id("page-dashboard")).to_be_visible()
        return self

DashboardPage — KPIs & modal

class DashboardPage:
    def kpi_value(self, name: str) -> Locator:
        return self.page.get_by_test_id(f"kpi-{name}-value")

    def open_new_run_modal(self) -> Self:
        self.new_run_btn().click()
        expect(self.run_modal()).to_be_visible()
        return self

    def should_have_activity_items(self, count: int = 5) -> Self:
        expect(
            self.activity_list().locator('[data-testid^="activity-item-"]')
        ).to_have_count(count)
        return self

Boas práticas

  • Page Objects em pages/ — locators, ações e assertions isolados
  • get_by_test_id + data-testid — mesma convenção Cypress/Playwright
  • login_via_api em fixture autouse — specs sem repetir login UI
  • scope="class" — resposta API compartilhada sem N requests idênticos
  • validate_schema — contrato REST declarativo com jsonschema
  • page.route + expect_request — mock e spy de API na UI
  • Fixtures JSON — dados determinísticos (credentials, team-member)
  • Um cenário por test_* — falhas fáceis de diagnosticar
  • pytestmark + markers — filtrar smoke/regression/api no CI
  • cache_auth_token session autouse — token uma vez por run
  • [TC-xxxx] — IDs rastreáveis alinhados ao TestFlow
