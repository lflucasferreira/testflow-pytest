What is PyTest?

PyTest is a testing framework for Python — it discovers tests automatically, organizes reusable fixtures, and produces clear failure reports.

  • Convention: test_*.py files, Test* classes, test_* functions
  • Fixtures — setup/teardown with dependency injection by parameter name
  • Markers — @pytest.mark.smoke, regression, api, visual…
  • Plugin ecosystem: pytest-playwright, pytest-xdist, axe-playwright-python
  • In this project: PyTest orchestrates · Playwright runs browser and HTTP
Unlike unittest, PyTest does not require inheriting from base classes — plain test_* functions and native assert with rich failure introspection are enough.

Why PyTest + Playwright?

Native Python

Ideal for QA/dev teams who prefer Python over TypeScript.

Declarative fixtures

autouse, scope="session" — auth cache once per run.

Unified E2E + API

page for UI and api_request for REST contracts.

TestFlow mirror

Same coverage as testflow-cypress and testflow-playwright.

Your first test

# tests/auth/test_login.py
import pytest
from playwright.sync_api import Page

from pages.login_page import LoginPage
from support.config import DEMO_EMAIL, DEMO_PASSWORD

pytestmark = pytest.mark.regression

@pytest.fixture(autouse=True)
def visit_login(page: Page) -> None:
    LoginPage(page).visit()

class TestValidCredentials:
    @pytest.mark.smoke
    def test_logs_in_via_ui(self, page: Page) -> None:
        login = LoginPage(page)
        login.login_with(DEMO_EMAIL, DEMO_PASSWORD)
        login.should_redirect_to_dashboard()

PyTest — assert introspection

Key differentiator for QAs: readable failures without custom matcher boilerplate.

def test_kpi_runs_is_positive(page):
    text = DashboardPage(page).kpi_value("runs").inner_text()
    assert int(text) > 0          # pytest mostra valores reais
    assert "Demo User" in greeting  # diff de strings no terminal

users = [{"role": "admin"}, {"role": "viewer"}]
assert users[0]["role"] == "viewer" # AssertionError + diff side-by-side
  • Plain assert — no framework assertion API
  • Report shows left/right for dicts, lists, and strings
  • Combine with Playwright expect() for UI with auto-wait

@pytest.fixture(scope="session", autouse=True)
def cache_auth_token(playwright) -> str:
    api_ctx = playwright.request.new_context(base_url=BASE_URL)
    session = fetch_auth_token(api_ctx)
    write_token_cache(session["token"], session["email"])
    return session["token"]

@pytest.fixture
def api_request(playwright) -> APIRequestContext:
    ctx = playwright.request.new_context(base_url=BASE_URL)
    yield ctx
    ctx.dispose()

validate_schema — REST contract

Schemas in fixtures/schemas/ — auth-login, user, users-list…

from support.helpers.fixtures import read_fixture
from jsonschema import validate

def validate_schema(instance, schema_path):
    schema = read_fixture(schema_path)
    validate(instance=instance, schema=schema)

Page Object Model (POM)

Thin specs · Page Objects encapsulate the DOM · Self return for method chaining.

  • Test spec — business scenario, Given/When/Then, fixtures
  • Page Object — locators, actions, assertions

Anatomy — LoginPage

class LoginPage:
    def email_input(self) -> Locator:
        return self.page.get_by_test_id("login-email")

    def login_with(self, email: str, password: str) -> Self:
        self.fill_email(email)
        self.fill_password(password)
        self.submit()
        return self

    def should_redirect_to_dashboard(self) -> Self:
        expect(self.page).to_have_url(re.compile(r"/web/dashboard\.html"))
        expect(self.page.get_by_test_id("page-dashboard")).to_be_visible()
        return self

DashboardPage — KPIs & modal

class DashboardPage:
    def kpi_value(self, name: str) -> Locator:
        return self.page.get_by_test_id(f"kpi-{name}-value")

    def open_new_run_modal(self) -> Self:
        self.new_run_btn().click()
        expect(self.run_modal()).to_be_visible()
        return self

    def should_have_activity_items(self, count: int = 5) -> Self:
        expect(
            self.activity_list().locator('[data-testid^="activity-item-"]')
        ).to_have_count(count)
        return self

Best practices

  • Page Objects in pages/ — isolated locators, actions, and assertions
  • get_by_test_id + data-testid — same Cypress/Playwright convention
  • login_via_api in autouse fixture — specs without repeating UI login
  • scope="class" — shared API response without N identical requests
  • validate_schema — declarative REST contract with jsonschema
  • page.route + expect_request — API mock and spy in UI tests
  • JSON fixtures — deterministic data (credentials, team-member)
  • One scenario per test_* — easy-to-diagnose failures
  • pytestmark + markers — filter smoke/regression/api in CI
  • cache_auth_token session autouse — token once per run
  • [TC-xxxx] — traceable IDs aligned with TestFlow
